# MERIDIAN Artifact Guide

This guide explains how to interpret and use the artifacts generated by MERIDIAN across its various modes.

## Artifact Overview

Each MERIDIAN mode produces specific artifacts that capture decisions, analyses, and outputs. All artifacts include:

- **Unique ID**: UUID for tracking
- **Fingerprint**: SHA-256 hash for integrity verification
- **Timestamps**: Creation and modification times
- **Lineage**: Parent artifact dependencies

## Mode-Specific Artifacts

### Mode 0.5: Opportunity Discovery

#### OpportunityBacklog
Contains prioritized list of ML opportunities for the business problem.

**Key Fields:**
- `opportunities`: Array of opportunity objects
  - `id`: Unique identifier
  - `type`: Classification (predictive, optimization, anomaly_detection, etc.)
  - `description`: Detailed opportunity description
  - `priority`: HIGH/MEDIUM/LOW
  - `feasibility_score`: 0-100 rating
  - `business_impact`: Expected outcome

**Example Usage:**
```bash
meridian artifacts show --type OpportunityBacklog | jq '.opportunities[0]'
```

#### OpportunityBrief
Selected opportunity with detailed implementation plan.

**Key Fields:**
- `selected_opportunity`: The chosen opportunity from backlog
- `stakeholder_brief`: Executive summary
- `data_requirements`: Required data sources
- `success_criteria`: Measurable goals

### Mode 0: Exploratory Data Analysis

#### Mode0GatePacket
Comprehensive data profiling and quality assessment.

**Key Fields:**
- `data_summary`: Dataset statistics
  - `n_rows`, `n_cols`: Dataset dimensions
  - `memory_usage_mb`: Size in memory
- `column_profiles`: Per-column analysis
  - `dtype`: Data type
  - `missing_ratio`: Fraction of nulls
  - `unique_values`: Cardinality
  - `statistics`: Min/max/mean/std for numeric
- `data_quality_issues`: Identified problems
- `recommendations`: Suggested preprocessing

**Interpretation Tips:**
- Check `missing_ratio` > 0.3 for problematic columns
- Review `data_quality_issues` before modeling
- Use `column_profiles` for feature selection

### Mode 1: Decision Intelligence

#### DecisionIntelProfile
Strategic alignment and hypothesis framework.

**Key Fields:**
- `business_kpi`: Target business metric
- `hypotheses`: Array of testable hypotheses
  - `statement`: Hypothesis text
  - `testable`: Boolean feasibility
  - `data_requirements`: Needed for testing
- `gate_verdict`: GO/NO_GO/CONDITIONAL
- `risk_assessment`: Potential challenges

**Decision Criteria:**
- GO: All hypotheses testable, data available
- CONDITIONAL: Some limitations, proceed with caution
- NO_GO: Critical blockers identified

### Mode 2: Feasibility Study

#### FeasibilityReport
ML viability assessment with statistical validation.

**Key Fields:**
- `dataset_stats`: Core statistics
  - `n_samples`, `n_features`: Dimensions
  - `target_distribution`: Class balance
- `feasibility_checks`: Validation results
  - `sufficient_data`: Boolean
  - `target_leakage_risk`: HIGH/MEDIUM/LOW
  - `class_balance_ok`: Boolean
- `ml_readiness_score`: 0-100 overall rating
- `recommendation`: PROCEED/CAUTION/STOP

**Red Flags:**
- `ml_readiness_score` < 60
- `target_leakage_risk` = HIGH
- `class_balance_ok` = false

### Mode 3: Strategy

#### ModelRecommendations
Algorithmic approach and model selection.

**Key Fields:**
- `primary_approach`: Main model type
  - `algorithm`: Specific algorithm
  - `complexity`: LOW/MEDIUM/HIGH
  - `interpretability`: Score 0-100
- `alternatives`: Backup approaches
- `feature_importance_method`: SHAP/permutation/etc.
- `validation_strategy`: Cross-validation approach

#### FeatureRegistry
Comprehensive feature catalog and engineering specs.

**Key Fields:**
- `features`: Array of feature definitions
  - `name`: Feature identifier
  - `dtype`: Data type
  - `source`: Original column(s)
  - `transformation`: Applied processing
  - `importance_score`: Predictive value
- `feature_groups`: Logical groupings
- `engineering_pipeline`: Processing steps

### Mode 4: Business Case

#### BusinessCaseScorecard
Quantified business value and ROI analysis.

**Key Fields:**
- `metrics`: Performance indicators
  - `accuracy`, `precision`, `recall`: ML metrics
  - `business_value`: Monetary impact
  - `roi_estimate`: Return on investment
- `cost_analysis`: Implementation costs
- `risk_factors`: Potential issues
- `recommendation`: Investment decision

#### ThresholdFramework
Decision boundaries and operating points.

**Key Fields:**
- `default_threshold`: Baseline cutoff
- `threshold_scenarios`: Different operating points
  - `threshold`: Probability cutoff
  - `precision`, `recall`: Expected performance
  - `business_impact`: Outcome at this threshold
- `optimization_criterion`: What to optimize for

### Mode 5: Code Generation

#### CodeGenerationPlan
Blueprint for generated ML pipeline.

**Key Fields:**
- `architecture`: System design
  - `components`: Pipeline modules
  - `dependencies`: Required packages
- `implementation_steps`: Development phases
- `output_structure`: Generated file layout
- `testing_strategy`: Validation approach

**Generated Output:**
Creates `PROJECT/` directory with:
- `src/pipeline.py`: Core ML implementation
- `demo.py`: Runnable demonstration
- `requirements.txt`: Dependencies

### Mode 6: Execution & Operations

#### ExecutionOpsScorecard
Production readiness assessment.

**Key Fields:**
- `code_quality`: Static analysis results
  - `pylint_score`: Code quality rating
  - `test_coverage`: Percentage covered
- `performance_tests`: Runtime benchmarks
  - `training_time_seconds`: Model training duration
  - `inference_latency_ms`: Prediction speed
- `operational_readiness`: Deployment checklist
- `issues_found`: Problems to address

### Mode 6.5: Interpretation

#### InterpretationPackage
Model explanations and business insights.

**Key Fields:**
- `model_interpretation`: Global explanations
  - `feature_importances`: Ranked features
  - `decision_rules`: Extracted logic
- `sample_predictions`: Example cases with explanations
- `bias_analysis`: Fairness assessment
- `business_narrative`: Executive summary

**Use Cases:**
- Stakeholder communication
- Regulatory compliance
- Model debugging

### Mode 7: Delivery

#### DeliveryManifest
Final package inventory and deployment guide.

**Key Fields:**
- `deliverables`: List of output files
  - `type`: Document/model/code
  - `path`: File location
  - `description`: Content summary
- `deployment_instructions`: Step-by-step guide
- `maintenance_plan`: Ongoing support
- `success_metrics`: KPIs to monitor

## Working with Artifacts

### Viewing Artifacts

```bash
# Show latest artifact of a type
meridian artifacts show --type FeasibilityReport

# View specific artifact
meridian artifacts show --id <uuid>

# Extract specific field with jq
meridian artifacts show --type ModelRecommendations | jq '.primary_approach'
```

### Verifying Integrity

```bash
# Verify all artifacts
meridian artifacts list --verify

# Check specific artifact
meridian artifacts show --id <uuid> --verify
```

### Artifact Dependencies

View artifact lineage:
```bash
# See what artifacts a mode requires
meridian mode 3 run --help  # Shows required artifacts

# Check parent artifacts
meridian artifacts show --type FeatureRegistry | jq '.parent_artifacts'
```

## Best Practices

1. **Always verify artifacts** before critical decisions
2. **Review gate verdicts** to understand mode outcomes
3. **Check parent artifacts** to understand dependencies
4. **Export important artifacts** for external use:
   ```bash
   meridian artifacts show --type BusinessCaseScorecard > business_case.json
   ```

5. **Monitor artifact size** - large artifacts may indicate issues:
   ```bash
   ls -lh .meridian/artifacts/mode_*/*.json
   ```

## Troubleshooting

### Missing Artifacts
If an artifact is missing:
1. Check if prerequisite modes completed: `meridian status`
2. Verify parent artifacts exist
3. Re-run the mode with `--headless` for faster execution

### Corrupted Artifacts
If fingerprint verification fails:
1. Check for manual edits to JSON files
2. Re-run the mode to regenerate
3. Check disk space and permissions

### Understanding Failures
When a mode fails:
1. Check the last artifact generated
2. Review `gate_verdict` and `recommendations`
3. Examine `data_quality_issues` or `risk_factors`

## Advanced Usage

### Custom Artifact Analysis

```python
import json
from pathlib import Path

# Load all artifacts of a type
artifacts_dir = Path(".meridian/artifacts")
feasibility_reports = []

for f in artifacts_dir.glob("*/FeasibilityReport_*.json"):
    with open(f) as file:
        feasibility_reports.append(json.load(file))

# Analyze trends
ml_scores = [r["ml_readiness_score"] for r in feasibility_reports]
print(f"Average ML Readiness: {sum(ml_scores)/len(ml_scores):.1f}")
```

### Artifact Export Pipeline

```bash
#!/bin/bash
# Export key artifacts for reporting

mkdir -p exports

# Export latest artifacts
for type in BusinessCaseScorecard FeasibilityReport ModelRecommendations; do
    meridian artifacts show --type $type > exports/${type}.json
done

# Create summary report
echo "# MERIDIAN Artifact Summary" > exports/summary.md
echo "Generated: $(date)" >> exports/summary.md
meridian status >> exports/summary.md
```

## Artifact Retention

Artifacts are stored in `.meridian/artifacts/mode_*/` directories. Configure retention in `meridian.yaml`:

```yaml
artifacts:
  retention_days: 30  # Auto-cleanup after 30 days
  auto_verify: true   # Verify on access
```

Manual cleanup:
```bash
# Remove artifacts older than 7 days
find .meridian/artifacts -name "*.json" -mtime +7 -delete
```
